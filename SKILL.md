# 客户支持组新品培训文档

> **前置条件:** 先阅读 [`../lark-shared/SKILL.md`](../lark-shared/SKILL.md)(认证/身份/权限)。
> 底层文档读写依赖 [`../lark-doc/SKILL.md`](../lark-doc/SKILL.md),画板依赖 [`../lark-whiteboard/SKILL.md`](../lark-whiteboard/SKILL.md),开权限依赖 [`../lark-drive/SKILL.md`](../lark-drive/SKILL.md)。

这份 skill 服务一个目标:让**研发/产品经理**写出**客户支持组(售后/客服)看得懂、用得上**的新品培训文档。判断标准始终是「**一线专员遇到客户提问时能不能直接照着答**」,而不是「技术上是否准确」。

> **完整细则与好例/反例案例** 见内部飞书《客户支持组培训文档编写指引》(含本公司具体案例)。需向客户支持组索取文档访问权限后,用 `lark-cli docs +fetch` 读取。本表是其抽象框架,具体案例以内部文档为准。

## 两个动作

| 动作 | 触发 | 做什么 |
|------|------|--------|
| **生成** | "给 X 产品写客服培训文档" / "出个培训文档模板" | 按 11 章模板生成飞书 docx。资料齐 → 填充草稿;资料缺 → 输出骨架 + 每章 TODO 提示 |
| **审查** | "检查这份培训文档" / 给出已有文档 URL | `docs +fetch` 读取 → 对照 11 章逐项打分 → 输出体检报告(缺章 / 踩反例 / 权限) |

---

## 11 章模板规范(核心知识)

章节顺序固定。`必填` = 任何新品都要写;`备选` = 视产品决定。每章的「红线」是最常见的反例,审查时逐条比对。

| # | 章节 | 必填 | 写作要点(好例) | 红线(反例,审查必查) |
|---|------|:----:|----------------|----------------------|
| 0 | **版本记录** | 必填 | 表格:版本号 / 修订内容 / 修订人 / 更新时间 | 没有版本表,后续没法追踪谁改了什么 |
| 1 | **产品定位** | 必填 | 一句话讲清定位/定义/用途/立项背景,让专员有整体认知;配图 | 没写 ❌ / 废话铺垫一大段 ❌ / 直接丢产品规格参数当定位 ❌ |
| 2 | **工作原理** | 备选 | 解释产品特性。**带"全球首个"定语或采用新机理的产品必写**;常见家电(风扇/灯泡)可跳过。**有图有对比**,优先用画板梳理原理 | 纯文字且关键原理点表述模糊 ❌ / 专业术语堆砌 ❌ / 废话多 ❌ |
| 3 | **主要功能** | 必填 | 功能/卖点/优势等独特性内容,**穿插功能实现原理**;有图有视频有说明,直观 | 纯表格罗列,理解困难 ❌ / 出现未解释的生态/专业名词 ❌ |
| 4 | **竞品对比 / 前代对比** | 必填 | 让专员快速答出"相比别家强在哪""客户买新款的理由";**高亮差异点、有前后对比** | 对比表里没有我家产品 ❌ / 罗列竞品功能但客服根本用不到 ❌ / 没重点(给价格标黄但无意义)❌ |
| 5 | **包装 / 配件一览** | 必填 | 清楚列出盒内有什么、没什么、哪些售卖哪些不售卖;**有图** | 纯文字看了等于没看 ❌ / 没写 → 客户问缺件,专员都不知是哪个 ❌ |
| 6 | **配件料号汇总表** | 必填 | 哪些配件能直接发客户、什么情况发单件或组合;**标清不同形态料号的区别**;系列产品按形态区分配件 | 纯文字需二次判断是哪个部件 ❌ / 料号与实际成品不一致,发出的配件客户无法使用 ❌ |
| 7 | **说明书 / 产品规格书** | 必填 | 规格表**直接粘贴进文档**并重点标注,而不是只给链接(现场不便点开评论);标清哪份是售后要看的 | 一堆文档不标重点 ❌ / 丢 PRD(售后用不到且常没开权限)❌ |
| 8 | **FAQ** | 必填 | 每条 = **问题 + 原因 + 解决方案 + 示意图/视频**;同步已知问题、收集新问题 | 权限没开 ❌ / 没写 ❌ / 收录用户根本遇不到的问题(复现要极复杂步骤——避免"程序员找 Bug"思维)❌ |
| 9 | **上线时间 / 未来规划 / 预计售价** | 备选 | 按事实填:今后支持什么、后续优化什么、面向哪些市场;**未知或不确定就不写** | 把没定的规划写成承诺 ❌ |
| 10 | **培训反馈** | 必填 | 放飞书问卷二维码,培训结束让专员扫码填写 | 没有反馈入口 ❌ |

**收尾铁律:** 文档写完**必须给客户支持组开阅读权限**。模板里反复强调——FAQ/规格书没开权限是最高频的反例。审查时务必确认权限已开。

## 生成:产品资料 → 飞书 docx

### Step 1 — 收资料,判断完整度
向研发要这些输入(缺什么记下来,对应章节生成 TODO):
- 产品名 / 定位一句话 / 是否带"全球首个"等新机理(决定第 2 章工作原理写不写)
- 主要功能与卖点、对应实现原理、可用图/视频
- 竞品或前代型号 + 差异点
- 包装清单(盒内/选购件)、配件料号(标清不同形态/组合)
- 规格书内容、FAQ 列表
- 上线时间/规划/售价(可选)、培训反馈问卷链接

### Step 2 — 生成文档
用 [`lark-doc`](../lark-doc/SKILL.md) 的 `docs +create`。标题统一格式 `【新品培训】<产品名>`:

```bash
lark-cli docs +create --api-version v2 --content '<title>【新品培训】产品名</title>...'
```

内容组织规则:
- **每章一个 H1 标题**,顺序严格按上表 0→10。
- 资料齐 → 直接写成段落/表格/图。资料缺 → 写章节标题 + 一行 `> ⚠️ 待研发补充:<具体要什么>`,**绝不留空白章节**。
- 第 0 章用表格;第 6 章料号、第 4 章对比用表格并**高亮差异**。
- **图片**:研发提供图后用 `docs +media-insert` 插入(截图优先 `--from-clipboard`)。
- 写作时套用上表「红线」自检,不要踩反例。

### Step 3 — 工作原理用画板(若第 2 章需要写)
原理类内容(信号路径、触发机理、模块关系)用画板比文字清楚。按 lark-doc 规则:
- 简单原理图:文档内直接嵌 `<whiteboard type="svg">完整 SVG</whiteboard>`。
- 复杂图:先建 `<whiteboard type="blank"></whiteboard>`,再读 [`lark-whiteboard`](../lark-whiteboard/SKILL.md) 用 DSL/Mermaid 写入。

### Step 4 — 开权限 + 交付
- 给客户支持组成员/群开**阅读**权限(收尾铁律),走 [`lark-drive`](../lark-drive/SKILL.md):
  ```bash
  lark-cli drive permission.members create --token <doc_token> --type docx \
    --data '{"member_type":"openchat","member_id":"<客服群chat_id>","perm":"view"}'
  ```
- 把文档 URL 和「哪些章节是 TODO」一并回给研发。

## 审查:已有文档 → 体检报告

### Step 1 — 读取
```bash
lark-cli docs +fetch --api-version v2 --doc "文档URL或token"
```

### Step 2 — 对照 11 章打分
逐章核对,输出报告:

```
## 培训文档体检报告 — <文档标题>

### 章节齐全度
| # | 章节 | 状态 | 问题 |
|---|------|------|------|
| 1 | 产品定位 | ✅/⚠️/❌ | ... |
...(必填章节缺失记 ❌,备选章节缺失记「合理跳过」)

### 踩了哪些红线
- 第 6 章料号:不同形态料号没区分,有发错件风险 ❌
- 第 8 章 FAQ:缺示意图,且疑似未开权限 ❌
...

### 权限检查
- [ ] 客户支持组阅读权限是否已开?(模板收尾铁律)

### 修改优先级
1. (阻塞)... 2. (重要)... 3. (优化)...
```

### 判定基调
- 必填章节缺失 = 不合格。
- 反例红线命中 = 标问题,给可操作改法(例:"把这段纯文字配件清单换成带图表格")。
- 始终回到判断标准:**一线专员能不能照着直接答客户**。

---

## 权限

| 操作 | 所需 scope |
|------|-----------|
| 创建/写入培训文档 | `docx:document:create`、`docx:document:write_only` |
| 读取已有文档(审查) | `docx:document:readonly`、`wiki:node:read`(wiki 链接时) |
| 插入图片/素材 | `docs:document.media:upload` |
| 创建画板节点 | `board:whiteboard:node:create` |
| 给客户支持组开阅读权限 | `docs:permission.member:create` |

> 写入是高风险操作前会触发确认门禁(exit 10),按 lark-shared「审批协议」处理:向用户确认后再加 `--yes`。
